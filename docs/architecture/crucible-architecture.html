Making One Session
Reliable

Crucible is one agent with station tools. The problem isn't "we need more agents." The problem is making a single Claude Code session do what it's told, with discipline, quality gates, and no shortcuts.

01

What Crucible Is

One Gemini supervisor with station tools. Each station is an agentrun process — Claude Code, Codex, OpenCode, or any supported CLI backend — scoped to one role. The operator controls what enters the line. One workpiece at a time. Sequential by design.

The Problem Being Solved

A single agent session (Claude Code, Codex, etc.) can turn intent into working code. But it skips review, doesn't verify, produces verbose output, and can't learn across sessions. Crucible wraps that session with planning discipline, quality gates, and operator control.

Operator Input Is Critical

Crucible's success depends on the quality of the GitHub issue. The operator writes the work order. The supervisor reads it, decomposes it, and dispatches to stations. A clear issue with acceptance criteria produces clean runs. A vague issue produces drift.

03

Five Measured Problems

1. Build → Build Chains

143 transitions (21% of all routing). The supervisor enters reactive loops without quality gates. In huli, 71% of builds had no review after them.

2. Ship Gets Denied 60%+

The supervisor tries to ship before the operator is ready. It can tell when code compiles — it can't tell when the operator considers the work complete.

3. Sessions Degrade After ~30 Dispatches

51% waste in mega-sessions. The supervisor loses coherence and falls into reactive loops. What should be multiple work orders gets jammed into one session.

4. Supervisor Doesn't Catch Architecture Problems

Built tools with raw sqlc when the project uses handler services. Never raised the concern. Executes plans without questioning whether the approach is right.

5. Clean Pipeline Sessions Work Well

Bounded tasks with clear issues: 0% waste, full pipeline, PRs shipped. The system works when scope is bounded and the supervisor follows the pipeline.

04

What Gets Caught, What Gets Missed

Stations Catch

  • Compilation failures and test regressions (verify)
  • Race conditions, nil panics, missing tests (review)
  • SDK type mismatches, missing error handling (inspect)
  • Wrong file placement, plan deviations (supervisor self-correction)

Stations Miss

  • Components built but never wired end-to-end
  • Config state not refreshing after writes
  • Token metadata silently dropped (found by external Codex reviewer)
  • Plugin ordering bugs making features dead at runtime
  • Architectural approach wrong for the codebase (raw sqlc vs handler services)
The Real Pipeline

Crucible produces draft-quality PRs. The operator merges, tests live, finds integration bugs, and feeds them back as new work orders. External AI reviewers (Codex, Gemini Code Assist) catch code-level bugs from a different model perspective. The operator is the integration test layer.

Crucible: draft → inspect → build → review → verify → ship (PR) ↓ Operator: merge → test live → find integration bugs → fix or new work order ↓ External: Codex, Gemini Code Assist catch what both missed

When It's Worth It

  • Size S-M task with a clear GitHub issue and acceptance criteria
  • Operator provides a work order and reviews the PR after
  • Project has test coverage (verify needs tests to run)
  • Codebase is well-structured (stations need conventions to follow)

When It's Overhead

  • Quick fix a single Claude Code session handles in 2 minutes
  • Exploratory work with no clear deliverable
  • Tasks requiring deep architectural judgment
  • Sessions that would exceed ~30 dispatches (split the work order)
06

What to Fix

Based on measured problems, not theoretical improvements. The data says: enforce the pipeline, improve operator input, keep sessions bounded.

Enforce the pipeline

The supervisor skips review 52% of the time. Routes should be enforced, not suggested. Plan-gated dispatch (#37), assembly line routing (#49), circuit breaker (#68). The supervisor can choose which route, but once chosen, every station in that route must run.

Improve operator input

Crucible's success correlates with issue quality. The blueprint artifact (#22) formalizes what a good work order looks like. Clear intent, acceptance criteria, constraints. The gap between "new work order gh#X" and what stations receive is the blueprint's job.

Track outcomes

The line doesn't remember. Outcome tracking (#51) records structured results from every dispatch. Prompt versioning (#52) makes station prompts evolvable. Without data, there's nothing to learn from.

Close the feedback loop

The Gödel loop (#50). Supervisor reads outcome history, proposes prompt changes, A/B tests them. The line gets smarter over time. Requires all three above to be in place first.
